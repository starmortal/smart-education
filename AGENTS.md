# 项目开发工作流程规范

## 📋 项目概述
本项目为智慧教育平台，包含前端（Vue.js）和后端（Node.js）两部分。

### 技术栈明细

#### 前端技术栈
- **框架**: Vue 3.2.13
- **UI 组件库**: Element Plus 2.13.0
- **路由**: Vue Router 4.6.4
- **HTTP 客户端**: Axios 1.13.2
- **图表库**: ECharts 6.0.0
- **Markdown 解析**: Marked 12.0.2
- **构建工具**: Vue CLI 5.0

#### 后端技术栈
- **框架**: Express 4.18.2
- **数据库**: MongoDB (Mongoose 7.0.0)
- **认证**: JWT (jsonwebtoken 9.0.3)
- **跨域**: CORS 2.8.5
- **环境变量**: dotenv 16.0.3
- **邮件服务**: Nodemailer 7.0.13
- **文件上传**: Multer 2.1.1
- **AI 集成**: OpenAI 4.0.0
- **日志系统**: Winston 3.x

## 🎯 核心开发原则

### 1. 代码质量原则
- 编写清晰、可维护的代码
- 遵循项目现有的代码风格和命名规范
- 添加必要的注释，特别是复杂逻辑部分
- 避免重复代码，提取公共逻辑

### 2. 安全原则
- 敏感信息（API密钥、数据库密码等）必须存放在 .env 文件中
- .env 文件已在 .gitignore 中忽略，不得提交到版本控制
- 用户输入必须进行验证和过滤
- 密码必须加密存储

### 3. 版本控制原则
- 不提交 node_modules、.env 等文件
- 遵循 .gitignore 规则

### 4. 文档维护原则
- 重要功能变更必须更新文档
- 遵循 `docs/AGENTS.md` 中的更新日志规范
- 在当前版本文档中持续追加更新内容
- 只有收到明确指令才创建新版本

### 5. 组件复用与性能原则
- **优先使用组件库实现**：能通过组件实现的功能，优先使用组件而非手写代码
- **不局限于已安装组件**：如果现有组件库无法满足需求，主动推荐引入新的组件库或开源框架
- **减少重复造轮子**：优先寻找成熟的开源解决方案，避免从零开发
- **主动推荐方案**：
  - 评估需求是否有现成的组件或库可用
  - 对比多个方案的优劣（如适用）
  - 推荐最适合的解决方案并说明理由
  - 如需引入新依赖，说明安装方法和基本用法
- **性能优先**：使用成熟组件库可以减少代码量、提高性能、保证兼容性、加快开发速度
- **用户引导**：如果用户不熟悉推荐的组件或库，主动提供：
  - 组件/库的简介和优势
  - 基本使用方法和代码示例
  - 安装命令（如需要）
  - 相关文档链接（如需要）

#### 已安装的组件和工具
**Element Plus 组件**：
- 表单组件：el-form, el-input, el-select, el-date-picker 等
- 数据展示：el-table, el-pagination, el-card, el-tag 等
- 导航组件：el-menu, el-tabs, el-breadcrumb 等
- 反馈组件：el-dialog, el-message, el-notification, el-loading 等
- 布局组件：el-container, el-row, el-col 等

**ECharts**：
- 用于数据可视化和图表展示
- 支持折线图、柱状图、饼图、雷达图等

**其他已安装工具**：
- Axios：HTTP 请求
- Vue Router：路由管理
- Marked：Markdown 渲染
- Multer：文件上传（后端）
- Nodemailer：邮件服务（后端）

#### 推荐新组件的原则
- 优先推荐 Vue 3 兼容的组件
- 选择活跃维护、文档完善的开源项目
- 考虑包体积和性能影响
- 评估学习成本和开发效率的平衡

## 🔄 开发工作流程

### 阶段一：需求确认（必须）
1. 仔细阅读用户需求
2. **如需求不明确，必须先询问澄清，不得直接开始开发**
3. 确认涉及的模块和文件范围
4. 与用户确认需求理解是否正确

### 阶段二：方案设计（必须）
1. 查看相关现有代码
2. 了解当前实现方式和代码结构
3. 识别需要修改的文件
4. 评估改动影响范围
5. **评估组件库和开源方案**
   - 优先考虑是否有现成的组件或库可以实现功能
   - 检查已安装的组件库（Element Plus、ECharts 等）是否满足需求
   - 如已安装组件不满足，搜索并推荐合适的新组件库或开源框架
   - 对比手写代码 vs 使用组件的优劣
6. **给出详细的执行方案，包括：**
   - 需要修改的文件列表
   - 主要改动内容说明
   - 推荐使用的组件或库（已安装或需新安装）
   - 使用组件方案的优势说明
   - 如需引入新依赖，提供安装命令和理由
   - 可能的影响范围
7. **如果用户不熟悉推荐的组件，主动提供：**
   - 组件/库的简介和为什么推荐它
   - 基本用法说明和代码示例
   - 安装方法（如需要）
   - 相关文档链接（如需要）
8. **等待用户确认方案后再开始执行**

### 阶段三：代码实现
1. 按照现有代码风格编写
2. 保持代码简洁，避免过度设计
3. 确保向后兼容，不破坏现有功能
4. 前后端接口保持一致
5. 优先使用组件库和开源方案

### 阶段四：测试验证
1. 检查代码语法错误
2. 确认逻辑正确性
3. 考虑边界情况和异常处理
4. 验证不影响其他功能

### 阶段五：文档更新（必须）
1. 更新 `docs/changelog/` 中当前版本的更新日志
2. 按分类记录改动内容
3. 更新日期为当前日期
4. 不添加任何 AI 相关标识

### 阶段六：工作报告（必须）
1. 给用户提供简洁的工作报告
2. 报告内容包括：
   - 完成的主要工作
   - 修改的文件列表
   - 需要注意的事项（如有）
3. **不要生成额外的说明文档或 markdown 文件**
4. 报告简洁明了，2-5句话即可

## 🔍 ESLint 配置说明

### 前端 ESLint 规则
项目采用平衡的 ESLint 配置，既保证代码质量，又不影响开发效率：

#### 配置模式
- **lintOnSave**: `'warning'` - 只在有错误时阻止编译，警告不会阻断

#### 规则说明
- `no-console`: `off` - 允许使用 console.log，方便调试
- `no-debugger`: `warn` - debugger 语句仅警告，不阻止编译
- `no-unused-vars`: `warn` - 未使用的变量仅警告
- `vue/multi-word-component-names`: `off` - 允许单词组件名
- `vue/no-unused-components`: `warn` - 未使用的组件仅警告
- `no-undef`: `error` - 未定义的变量报错（防止拼写错误）
- `no-unreachable`: `error` - 不可达代码报错
- `no-dupe-keys`: `error` - 重复的对象键报错

#### 修改规则
如需调整规则，编辑 `frontend/package.json` 中的 `eslintConfig.rules` 部分。

## 📊 日志系统说明

### 后端日志配置
项目使用 Winston 日志系统，提供统一的日志记录：

#### 日志级别
- `error`: 错误信息（记录到 error.log）
- `warn`: 警告信息
- `info`: 一般信息
- `debug`: 调试信息

#### 日志文件
- `logs/error.log`: 错误日志（最大 5MB，保留 5 个文件）
- `logs/combined.log`: 所有日志（最大 5MB，保留 5 个文件）

#### 使用方法
```javascript
const logger = require('./utils/logger');

logger.info('操作成功');
logger.warn('警告信息');
logger.error('错误信息', error);
```

#### 环境配置
- 开发环境：同时输出到控制台和文件
- 生产环境：仅输出到文件

## 🔄 API 响应格式规范

### 统一响应格式
所有 API 接口返回统一的 JSON 格式：

```javascript
{
  "code": 200,           // 状态码
  "message": "操作成功",  // 提示信息
  "data": {},            // 返回数据（可为 null）
  "timestamp": 1234567890 // 时间戳
}
```

### 使用响应工具类
```javascript
const Response = require('./utils/response');

// 成功响应
Response.success(res, data, '操作成功');

// 错误响应
Response.error(res, '操作失败', 500);
Response.badRequest(res, '参数错误');
Response.unauthorized(res, '未授权');
Response.notFound(res, '资源不存在');

// 分页响应
Response.page(res, list, total, page, pageSize);
```

### 错误处理
使用统一的错误处理中间件和自定义错误类：

```javascript
const { AppError, asyncHandler } = require('./middleware/errorHandler');

// 抛出自定义错误
throw new AppError('错误信息', 400, 'ERROR_CODE');

// 包装异步路由
router.get('/path', asyncHandler(async (req, res) => {
  // 异步代码，错误会自动被捕获
}));
```
