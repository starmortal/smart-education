# 更新日志工作规范

## ⚠️ 重要规则

### 版本更新原则
- **未收到明确指令前，不得新建版本号和新文档**
- 在当前最新版本号的文档中持续追加和整理更新内容
- 只有在收到明确的"发布新版本"或"创建新版本"指令后，才能创建新的版本文档

### 内容编写要求
- **必须填写准确的日期**（格式：YYYY-MM-DD）
- **严禁在文档底部或任何位置添加"AI 生成"、"由 AI 编写"等相关字眼**
- 保持内容专业、简洁、清晰

## 📁 文件结构
```
docs/
└── changelog/
    ├── v1.0.0.md
    ├── v1.0.1.md
    ├── v1.1.0.md
    └── v2.0.0.md
```

## 🔢 版本号规则（语义化版本）

版本号格式：`vX.Y.Z`

- **X (主版本号)**: 重大更新，可能包含不兼容的 API 变更
  - 示例：v1.0.0 → v2.0.0
  - 触发条件：架构重构、重大功能变更、破坏性更新

- **Y (次版本号)**: 功能更新，向下兼容
  - 示例：v1.0.0 → v1.1.0
  - 触发条件：新增功能模块、新增页面、重要功能优化

- **Z (修订号)**: Bug 修复和小优化，向下兼容
  - 示例：v1.0.0 → v1.0.1
  - 触发条件：Bug 修复、UI 微调、性能优化、文案修改

## 📝 更新日志编写规范

### 文件命名
- 文件名：`vX.Y.Z.md`
- 位置：`docs/changelog/` 目录下
- 示例：`v1.0.1.md`, `v1.2.0.md`, `v2.0.0.md`

### 分类说明

#### ✨ 功能新增
记录所有新增的功能模块、页面、接口等
- 格式：`[模块/页面名称] 具体功能描述`
- 示例：
  - `[用户中心] 新增个人资料编辑功能`
  - `[学习计划] 新增 AI 智能推荐学习路径`
  - `[题库系统] 新增错题自动归类功能`

#### 🎨 UI优化
记录界面、样式、布局的优化改进
- 格式：`[页面/组件名称] 优化描述`
- 示例：
  - `[登录页] 优化表单布局，提升移动端体验`
  - `[导航栏] 调整配色方案，增强视觉层次`
  - `[卡片组件] 统一圆角和阴影样式`

#### 🐛 Bug修复
记录所有 Bug 修复
- 格式：`[模块名称] 修复具体问题`
- 示例：
  - `[用户登录] 修复验证码刷新失败的问题`
  - `[错题本] 修复题目收藏状态不同步的 Bug`
  - `[数据统计] 修复图表数据显示错误`

#### ⚡ 性能优化
记录性能相关的优化
- 示例：
  - `优化首页加载速度，减少 30% 加载时间`
  - `优化数据库查询，提升列表渲染性能`
  - `实现图片懒加载，降低首屏加载压力`

#### 🔧 依赖更新
记录依赖包的更新和变更
- 格式：`更新/新增 包名 版本号`
- 示例：
  - `更新 Vue 至 3.4.0`
  - `更新 Element Plus 至 2.5.0`
  - `新增 dayjs 1.11.0 用于日期处理`

#### 📦 组件引入
记录新引入的组件库或第三方组件
- 格式：`引入 组件库名称 版本号 - 用途说明`
- 示例：
  - `引入 vue-draggable 4.1.0 - 实现拖拽排序功能`
  - `引入 @vueuse/core 10.0.0 - 提供常用组合式函数`
  - `引入 vxe-table 4.5.0 - 增强表格功能`

#### 🎯 体验优化
记录用户体验、交互流程的改进
- 示例：
  - `优化表单提交流程，增加加载提示`
  - `改进错误提示文案，更加友好易懂`
  - `优化页面切换动画，提升流畅度`
  - `增加操作引导提示，降低学习成本`

#### 🔐 安全更新
记录安全相关的修复和更新
- 示例：
  - `修复 XSS 漏洞`
  - `加强密码加密强度`
  - `更新依赖包修复安全漏洞`
  - `增加接口访问频率限制`

#### 📝 文档更新
记录文档相关的更新
- 示例：
  - `更新 API 文档`
  - `完善开发指南`
  - `新增部署文档`

#### 🗑️ 功能移除
记录移除的功能及原因
- 格式：`移除 功能名称 - 原因说明`
- 示例：
  - `移除旧版编辑器 - 已被新版本替代`
  - `移除未使用的依赖包 - 减小项目体积`

#### 🔨 重构优化
记录代码重构和架构调整
- 示例：
  - `重构用户模块，采用组合式 API`
  - `优化路由结构，实现懒加载`
  - `抽取公共逻辑，提高代码复用性`

#### 🧪 测试相关
记录测试相关的更新
- 示例：
  - `新增用户模块单元测试`
  - `提升测试覆盖率至 80%`
  - `修复测试用例失败问题`

#### 🔧 其他改动
记录配置、环境、工具等其他变更
- 示例：
  - `添加 .gitignore 忽略 .env 文件`
  - `更新 ESLint 配置规则`
  - `优化 Webpack 打包配置`

## 🤖 AI 助手操作流程

### 日常更新流程（默认操作）

1. **定位当前版本文档**
   - 找到 `docs/changelog/` 目录下最新的版本文档
   - 在该文档中追加新的更新内容

2. **整理更新内容**
   - 按照分类（功能新增、UI优化、Bug修复等）整理内容
   - 将新内容添加到对应分类下
   - 保持格式统一

3. **更新日期**
   - 确保文档顶部的发布日期为最新修改日期
   - 日期格式：YYYY-MM-DD

4. **内容要求**
   - 描述清晰、简洁
   - 使用中文
   - 每条更新独立成行
   - 重要更新可以添加详细说明
   - **严禁添加任何 AI 相关标识**

### 新版本发布流程（需明确指令）

**触发条件**：收到用户明确的"发布新版本"、"创建 vX.Y.Z 版本"等指令

1. **确认版本号**
   - 由用户明确指定新版本号
   - 如未指定，根据版本号规则建议版本号并等待确认

2. **创建新版本文档**
   - 在 `docs/changelog/` 目录下创建 `vX.Y.Z.md` 文件
   - 使用标准模板

3. **填写内容**
   - 填写准确的发布日期
   - 整理本版本的所有更新内容
   - 按分类组织

4. **版本号规则参考**
   - 有破坏性更新 → 主版本号 +1 (v2.0.0)
   - 有新功能 → 次版本号 +1 (v1.1.0)
   - 仅 Bug 修复/小优化 → 修订号 +1 (v1.0.1)

## 📌 注意事项
1. **默认在当前最新版本文档中追加内容，不要自行创建新版本**
2. 只有收到明确的版本发布指令后，才创建新版本文档
3. 版本号必须递增，不能跳跃或重复
4. **必须填写准确的日期**（格式：YYYY-MM-DD）
5. **严禁在文档任何位置添加 AI 生成相关字眼**
6. 更新内容要具体，避免模糊描述
7. 保持格式统一，便于阅读和维护
